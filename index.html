The MonoTable Data Structure

"Everything should be made as simple as possible, but no simpler." -Einstein

MonoTable stores all data in a single table. The table consists of records sorted by their keys. Each record, in addition to their key, can have 0 or more named fields. Basicaly, it's a 2-dimensional hash where the first dimension supports range selects.

Many User-Tables in the MonoTable

The MonoTable can be easily broken up into multiple "virtual-tables" by prefixing all keys in a virtual-table with it's name: table1/keyA, table2/keyZeta. MonoTable was designed to efficiently support this use-case. Keys are allowed to be large and common prefexes are stored efficiently.

Easily Scalable

Most NoSQL options claim to be scalable, but in fact most are not or at least are not easilly scalable. Monotable is designed to be scalable along each of the core metrics: reads/s, writes/s, and total data-size. Further, Monotable is designed to be "realtime" scalable. You can scale without ever taking the system offline.

Scaling is easy. Fire up another server. Point it at the Monotable cluster. Done.

The only practical limit for Monotable is the maximum record size (typically 32 megabytes). Otherwise there are no limits:

  • Maximum # of records: none
  • Maximum # of servers: none
  • Maximum # of bytes stored: none

Easy to Admin

  • Number of daemons/server: 1
  • Number of server types in cluster: 1
  • Single points of failure: none
  • Automatic server failure recovery
  • Automatic data corruption detection and recovery
  • Automatic load balancing
  • Data structure designed for easy manual recovery in the worst-case scenario

Competitive Survey

  • Hadoop: Complex to config and admin. NameNode - single point of failure, both for serving and data integrity. Scalability limited by NameNode memory.
  • Cassandra: No data-locality. Limited map-reduce due to unsorted data. Difficult to add and remove nodes.
  • CouchDB: Data scaling only via manual sharding.

Future

Once we get V1 down, we'd like to add support for a Map-Reduce. The core data-structure is designed to work well with Map-Reduce.
